Normative Assertions Summary

Web of Things (WoT) Thing Description

td-vocabulary: MUST
All vocabulary restrictions noted in these tables MUST be followed, including mandatory items and default values.
  • NO TEST SPECIFICATION
td-unique-identifiers: MUST
Each instance of a Property, Action, and Event class MUST have an identifier that is unique within the context of a Thing Description document.
  • NO TEST SPECIFICATION
td-jsonld-keywords: MAY
Thing Description instances MAY contain JSON-LD 1.1 keywords such as @context and @type.
    • TESTED
    • EXAMPLE EXISTS only for @context
    • COUNTER EXAMPLE CANNOT EXIST
td-string-type: MUST
Vocabulary terms that use simple types string and anyURI MUST be serialized as JSON string.
    • ASSERTION NOT CLEAR (does this mean that strings we have in TD such as "type":"number" must be "type" and not (type) or (type is number) etc.? )
    • IF YES THEN TESTED
    • EXAMPLE DOES NOT EXIST
    • COUNTER EXAMPLE DOES NOT EXIST
td-integer-type: MUST
Vocabulary terms that use simple type unsignedInt MUST be serialized as JSON integer.
    • ASSERTION NOT CLEAR (does this mean that strings we have in TD such as "type":123 must be 123 and not (1 2 3))
    • IF YES THEN TESTED
    • EXAMPLE DOES NOT EXIST
    • COUNTER EXAMPLE DOES NOT EXIST
td-number-type: MUST
Vocabulary terms that use simple type double MUST be serialized as JSON number.
    • ASSERTION NOT CLEAR (does this mean that strings we have in TD such as "type":123 must be 123 and not (1 2 3))
    • IF YES THEN TESTED
    • EXAMPLE DOES NOT EXIST
    • COUNTEREXAMPLE DOES NOT EXIST
td-context: MAY
The root object of a Thing Description instance MAY include the @context key from JSON-LD 1.1 with the value URI of the Thing description context file http://www.w3.org/ns/td.
    • ASSERTION NOT CLEAR: Can you have an @context key without this context file
    • IF YES THEN TESTED WRONGLY
td-context-jsonld: MUST
When a Thing Description instance is processed and interpreted by a JSON-LD 1.1 processor the @context field MUST be present
    • NOT TESTED
    • HOW TO TEST
td-additional-contexts: MUST
When a single Thing Description instance involves several contexts, additional namespaces with prefixes MUST be appended to the @context array structure.
    • TESTED
    • EXAMPLE EXIST BUT NOT EXPLICITELY DONE FOR THIS ASSERTION
td-context-toplevel: MUST
Each mandatory and optional field name as defined in the class Thing MUST be serialized as a JSON key in the root object of the Thing Description instance.
    • TESTED
    • EXAMPLE EXIST BUT NOT EXPLICITELY DONE FOR THIS ASSERTION
    • COUNTER EXAMPLE DOESN'T EXIST
td-objects: MUST
The type of the fields properties, actions, events, and version MUST be a JSON object.
    • TESTED
    • EXAMPLE EXIST BUT NOT EXPLICITELY DONE FOR THIS ASSERTION
    • COUNTER EXAMPLE DOESN'T EXIST
td-arrays: MUST
The type of the fields links, scopes, and security MUST be a JSON array.
    • TESTED
    • EXAMPLE EXIST BUT NOT EXPLICITELY DONE FOR THIS ASSERTION
    • COUNTER EXAMPLE DOESN'T EXIST
td-properties: MUST
Properties (and sub-properties) offered by a Thing MUST be collected in the JSON-object based properties field with (unique) Property names as JSON keys.
    • NOT TESTED
    • TDS THAT DON'T RESPECT THIS ASSERTION ARE VALIDATED
td-property-keys: MUST
Each mandatory and optional vocabulary term as defined in the class Property, as well as its two superclasses InteractionPattern and DataSchema, MUST be serialized as a JSON key within a Property object.
    • ASSERTION NOT CLEAR
td-property-objects: MUST
The type of the fields properties and items MUST be serialized as a JSON object.
    • ASSERTION NOT CLEAR (SOUNDS WRONG SINCE ITEMS CAN BE AN ARRAY IN JSON SCHEMA)
    • NOT TESTED
td-property-arrays: MUST
The type of the fields forms, required, and enum, scopes, and security, MUST be serialized as a JSON array.
    • TESTED
td-property-semantic: MAY
Similar to the case at the Thing level, properties MAY have additional semantic annotations based on JSON-LD 1.1 keywords.
    • CANNOT BE TESTED
    • EXAMPLE EXISTS
td-property-defaults: MUST
When a Thing Description instance is processed and interpreted by a JSON-LD 1.1 processor, each property MUST contain the vocabulary terms observable and readOnly due to the open-world assumption of Linked Data.
    • NOT TESTED
    • NOT CLEAR HOW TO TEST
td-actions: MUST

Every Thing MUST have at least one security scheme specified at the top level.

td-security-overrides: MAY
Security schemes MAY also be specified at the interaction and form levels. In this case, definitions at the lower levels override (completely replace) the definitions at the higher level.
  • NO TEST SPECIFICATION
td-security-binding: MUST
If a Thing requires a specific access mechanism for a resource, that mechanism MUST be specified in the Thing Description's security scheme configuration for that resource.
    • HOW TO TEST?
    • PENETRATION TEST?
td-security-no-extras: MUST NOT
If a Thing does not require a specific access mechanism for a resource, that mechanism MUST NOT be specified in the Thing Description's security scheme configuration for that resource.
  • NO TEST SPECIFICATION

Before starting JSON-LD 1.1 processing of a Thing Description instance, all external vocabulary terms used in the Thing Description MUST provide their context URIs as prefix or within the @context field.

td-sec-best-practices: SHOULD
When designing new devices and services for use with the WoT, we have documented a set of best practices in [[!WOT-SECURITY-BEST-PRACTICES]] that SHOULD be followed.
  • NO TEST SPECIFICATION
td-sec-mutable-ids: SHOULD
There SHOULD be a mechanism to update the id of a Thing.
  • NO TEST SPECIFICATION
td-sec-authorize-tds: SHOULD
Only authorized users SHOULD be provided access to the Thing Description for a Thing.
  • NO TEST SPECIFICATION
td-sec-authenticate-tds: SHOULD
Thing Descriptions SHOULD only be obtained through mutually authenticated channels.
  • NO TEST SPECIFICATION
td-sec-consent: SHOULD
A Thing Description associated with a personal device SHOULD be treated as if it contained personally identifiable information.
  • NO TEST SPECIFICATION
